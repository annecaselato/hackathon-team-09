# ADR-007: Audit Trail Continuity Across Legacy and Modern Systems

**Status**: Accepted  
**Date**: 2026-04-29  
**Deciders**: Vitor Filincowsky (Tech Lead), Danilo Lisboa (Backend Dev), Reges Alves (QA)

---

## 📖 Context

Stage 1 archaeology revealed critical audit trail facts:

- **AUDITORIA DDM (FNR 153)**: 25M+ immutable records since 2005
- **Legal requirement**: Lei 8159 mandates 10-year minimum retention of public records
- **Immutability**: No DELETE or UPDATE of audit records — database constraint enforced
- **Action codes**: IN (insert), AL (alter), EX (delete attempt), CO (confirm), DV (divergence), LG/LO (login/logout), BT (batch), ER (error), AU (audit), RE (reversal)
- **RELAUDIT.NSN (line 108)**: Standard report excludes EX actions — compliance teams must request "full audit" to see deletion attempts
- **COD-MODULO**: Source Natural program name stored in each audit record (CADBENEF, BATCHPGT, etc.)

**The problem**: When SIFAP 2.0 goes live, new audit records will be generated by Java classes (not Natural programs). The audit trail must maintain continuity across:
1. Historical records from legacy (Natural programs, COD-MODULO = CADBENEF, BATCHPGT, etc.)
2. New records from modern system (Spring services, COD-MODULO = BeneficiaryService, PaymentBatchJob, etc.)

Auditors and compliance teams must be able to query a unified, continuous audit trail spanning 1998 to present.

---

## ⚖️ Decision

We will maintain a **single unified audit table** in PostgreSQL, with all 25M+ migrated legacy records and all new records from SIFAP 2.0, with the following design:

### Audit Table Schema (PostgreSQL)

```sql
CREATE TABLE auditoria (
    num_auditoria     BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
    dt_evento         DATE NOT NULL,
    hr_evento         TIME NOT NULL,
    ts_evento         TIMESTAMPTZ NOT NULL DEFAULT now(),
    cod_acao          CHAR(2) NOT NULL CHECK (cod_acao IN ('IN','AL','EX','CO','LG','LO','BT','ER','AU','RE','DV')),
    cod_modulo        VARCHAR(50) NOT NULL,    -- legacy: 'CADBENEF'; modern: 'BeneficiaryService'
    tipo_entidade     CHAR(4) NOT NULL CHECK (tipo_entidade IN ('BENF','PGTO','PROG','ADMN','SIST')),
    id_entidade       VARCHAR(15) NOT NULL,
    campos_antes      JSONB,                   -- was MU GRP-ANTES in Adabas
    campos_depois     JSONB,                   -- was MU GRP-DEPOIS in Adabas
    usr_evento        VARCHAR(8) NOT NULL,
    nome_usuario      VARCHAR(40),
    cod_perfil        CHAR(3),
    ip_origem         INET,
    sistema_origem    CHAR(1) DEFAULT 'M'      -- 'L' = Legacy, 'M' = Modern (SIFAP 2.0)
) PARTITION BY RANGE (dt_evento);

-- Immutability: revoke all destructive permissions from app user
REVOKE DELETE, UPDATE, TRUNCATE ON auditoria FROM sifap_app;

-- Trigger blocks bulk delete even for DBA
CREATE RULE auditoria_no_delete AS ON DELETE TO auditoria DO INSTEAD NOTHING;
```

### Key Design Decisions

1. **`sistema_origem` field (new)**: `'L'` for migrated legacy records, `'M'` for modern system records — allows filtering while maintaining unified view
2. **`cod_modulo` preserves legacy program names**: Legacy records keep `CADBENEF`; modern records use Spring class name (`BeneficiaryService`)
3. **`campos_antes/depois` as JSONB**: Replaces Adabas MU (multiple value) fields GRP-ANTES/GRP-DEPOIS; stores old/new values as structured JSON
4. **Partitioning by `dt_evento`**: 29 years × 12 months = 348 partitions; enables pruning in compliance queries
5. **`GENERATED ALWAYS AS IDENTITY`**: PostgreSQL auto-increment replaces Adabas ISN; migrated records get sequential IDs starting from 25M+1

### Audit Service Design

```java
@Service
public class AuditService {

    // Maps to legacy STORE AUDITORIA-V pattern
    public void record(AuditEvent event) {
        Auditoria entity = Auditoria.builder()
            .tsEvento(Instant.now())
            .codAcao(event.action())      // AuditAction.INSERT, ALTER, etc.
            .codModulo(event.source())    // "BeneficiaryService", "PaymentBatchJob"
            .tipoEntidade(event.entityType())
            .idEntidade(event.entityId())
            .camposAntes(event.before())  // JSONB serialized
            .camposDepois(event.after())  // JSONB serialized
            .usrEvento(event.userId())
            .sistemaOrigem('M')          // Modern system
            .build();
        auditRepository.save(entity);
    }
}
```

## 💡 Rationale

### Why a Single Unified Table

- **Compliance simplicity**: Auditors query one table, not two systems, for compliance reports
- **Temporal continuity**: No gap in audit trail between legacy cutoff and modern go-live
- **Standard SQL queries**: `SELECT * FROM auditoria WHERE id_entidade = '12345678901' ORDER BY ts_evento` returns complete history
- **`sistema_origem` transparency**: Compliance teams can distinguish legacy vs. modern records when needed, without breaking unified queries

### Why JSONB for campos_antes/campos_depois

- Adabas GRP-ANTES/GRP-DEPOIS are MU (multiple value) fields — each can contain up to 20 field name/value pairs
- JSONB naturally represents this: `{"NOME": "João Antigo", "STATUS": "A"}` → `{"NOME": "João Novo", "STATUS": "S"}`
- Indexed JSONB allows `WHERE campos_antes @> '{"STATUS": "A"}'::jsonb` — equivalent to Adabas HISTOGRAM

### Why Preserve Legacy cod_modulo Values

- Migrated audit records say `CADBENEF`, `BATCHPGT` — these are legal records; must not be altered
- New records say `BeneficiaryService`, `PaymentBatchJob` — consistent with Java naming
- `sistema_origem = 'L'` or `'M'` allows auditors to understand which system generated each record

---

## 📊 Consequences

### Positive

- **Legal compliance**: Continuous audit trail from 2005 to present; no gap during cutover
- **Unified reporting**: Single RELAUDIT equivalent (RELAUDIT 2.0 API endpoint) queries all records
- **Immutability enforced**: PostgreSQL rule + revoked permissions enforce BR-AUD-001
- **Query performance**: Partitioning by date + JSONB indexes enable fast compliance queries

### Negative / Trade-offs

- **Storage**: Migrated 25M+ records (30GB) + ongoing growth (2M/year) in same table; partitioning manages this
- **JSONB vs. normalized**: Querying specific field changes in JSONB is less performant than normalized columns; mitigated by GIN indexes on JSONB
- **Legacy program names in modern table**: `COD-MODULO = 'CADBENEF'` is legacy; future queries must know both naming conventions

---

## 🔀 Alternatives Considered

### Alternative A: Separate Legacy Archive Table + New Modern Audit Table
- **Pros**: Clean separation, no naming confusion
- **Rejected because**: Compliance queries would need UNION across two tables; legal precedent requires demonstrating continuous single audit trail

### Alternative B: Event Sourcing with Kafka / Event Store
- **Pros**: Natural audit trail, replay capability, decoupled
- **Rejected because**: Existing 25M+ Adabas records would need event re-construction; adds Kafka infrastructure; team unfamiliar; overkill for current scope

### Alternative C: Azure Log Analytics for All Audit
- **Pros**: Azure-native, searchable, managed retention
- **Rejected because**: Cannot store legacy records in Log Analytics efficiently (cost); querying joins between Log Analytics and PostgreSQL is complex; legal requirement for SQL-queryable data

---

## 🛡️ Risk Mitigation

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Legacy audit records altered during migration | LOW | CRITICAL | ETL: compare SHA-256 hash of every migrated record vs. original; zero tolerance for changes |
| JSONB migration loses MU field semantics | MEDIUM | HIGH | Unit tests comparing Adabas MU→JSONB conversion for 5 known audit records with verified values |
| Immutability broken by DBA override | LOW | HIGH | `RULE auditoria_no_delete` + PostgreSQL audit log of all DDL/DML operations |
| Compliance audit fails due to naming discrepancy | LOW | MEDIUM | Document `sistema_origem` field and `cod_modulo` conventions in compliance handbook |
